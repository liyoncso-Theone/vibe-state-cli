## Loop closure (per `[[cross-team-dispatch-via-rfc]]`)

This document is the **adoption response** that closes the dispatch loop started by the laptop-side Claude session's RFC. The pattern in full:

1. **Dispatch**: laptop-side Claude wrote `docs/v0.3.6-rfc-agents-md-bm-aware.md` directly into the target repo (vibe-state-cli), did not commit code, did not modify existing files. Decision rights handed to owner.
2. **Receive**: owner (Theone) returned to vibe-state-cli on 2026-06-10; the RFC surfaced naturally because it was sitting in the repo's docs/ folder.
3. **Review**: owner + main-dev session went through 3 review-AI rounds (the laptop session's own reframe, then a separate review-AI's push-back, then a 4-reviewer adversarial workflow before commit).
4. **Decide**: owner explicitly chose to break the 90-day v0.3.6 freeze for urgent multi-agent need; chose adjusted Plan A.
5. **Respond** (this document): adopt + adjust + ship + write back. **The response doc is mandatory** — without it, the laptop session has no signal that its dispatch reached the maintainer, and the loop stays open.

If a future RFC arrives via the same dispatch pattern, the same closure document MUST accompany the ship.

## What was adopted as-is

- **Plan A (default-on, no flag)** — RFC's recommended path. Adopted.
- **Concrete name `"Basic Memory"` over abstract `"second brain"`** — adopted. Concrete naming matches the actual MCP tool names (`mcp__basic-memory__*`); future backend swaps update one template, not every consumer.
- **Section position: between `## Session Start` and `## Workflow`** — adopted as proposed.
- **Default-on (RFC Plan A core)** — adopted despite 2 of 4 adversarial reviewers voting BLOCK. The asymmetry vs v0.3.6's `[promotion].enabled = false` is explicitly documented in CHANGELOG as conscious design choice (outbound writes default off, inbound read-guidance defaults on for the assumed multi-agent setup).

## What was adjusted vs the RFC proposal

| RFC said | What shipped | Why |
|---|---|---|
| `projects = ["personal", "methodology"]` as default | `projects = []` as default | OSS leak risk. RFC's defaults are Theone's BM project names; pip-installed non-Theone users would see those names appear in their auto-generated AGENTS.md, which conflicts with vibe-state-cli's cross-agent-neutral marketing. Empty list renders a generic instruction ("query whichever projects you find"); explicit list is opt-in per workspace. |
| Fallback: "trust agent's own error handling" / "proceed without it" | Concrete fallback: name `.vibe/state/` baseline, prescribe a one-line warning shape, prohibit retries and blocking | Adversarial review (3 of 4 reviewers) independently flagged that "proceed without it" leaves agent behavior undefined. Cold-start risk (BM ~30s on Windows — the exact symptom that motivated the RFC) made vague fallback dangerous. |
| Open question 5 ("BM offline / MCP not registered — let agent handle it?") | Resolved: template explicitly tells agent the baseline + the warning shape + retry/block prohibition + cold-start cap | Same reasoning as above. |
| No version target ("v0.3.6 or later") | Shipped as v0.3.7 | v0.3.6 was already in 90-day post-ship freeze. Owner deliberately broke freeze for this; freeze clock resets from v0.3.7 ship date. |
| Implementation Notes: "small (~15 lines of template string)" | Actual: `MemorySection` config class + `_build_memory_section` + `_build_basic_memory_section` helpers + 10 new tests, ~150 lines total | The RFC underestimated because it didn't account for the vendor-neutral `target` dispatch, the OSS-safe empty-projects default rendering, or the offline fallback prose. Still small overall. |

## Resolutions of the RFC's 6 open questions

| # | RFC question | Resolution |
|---|---|---|
| 1 | Name: "Basic Memory" vs "second brain" | **"Basic Memory"** (concrete). Abstract names get stale fast; concrete naming matches the MCP tool names users will see in errors. Re-evaluate if BM is ever replaced. |
| 2 | Which command regenerates AGENTS.md? | **`vibe sync` and `vibe start`** (both call `refresh_adapters()` from v0.3.5). The post-commit hook does NOT (v0.3.6 deliberately decoupled hook path from current.md writes). |
| 3 | Project names hardcoded? | **No**. Defaults to `[]`; users opt in via `[memory].projects = ["foo", "bar"]` in their `.vibe/config.toml`. Owner's `personal` / `methodology` names live in owner's local configs, never in template defaults. |
| 4 | Link to specific BM notes? | **No**. Keeps the template tool-agnostic and avoids stale links. |
| 5 | What if BM is offline? | **Template specifies the full fallback** (see Adjustments table above). Not delegated to agent. |
| 6 | Update `claude` adapter's rules? | **No**. v0.3.6 already shimmed `.claude/rules/vibe-standards.md` to `See AGENTS.md` when `agents_md` is co-enabled. Slim-mode inheritance was already correct. |

## Adversarial review outcomes (compressed)

4 reviewers ran in parallel before commit (matches v0.3.6's pre-commit review pattern). 1 synthesizer consolidated.

| Reviewer dimension | Verdict |
|---|---|
| BM offline / MCP unreachable usability | ship-with-followup |
| Default-on vs opt-in discipline | ship-with-followup |
| Cross-agent assumption for non-owner users | **block** (refuted) |
| Template idempotency + migration | **block** (refuted) |
| **Synthesizer** | **SHIP-WITH-FOLLOWUP** (confidence 0.82) |

**Both BLOCK verdicts were refuted**:
- Idempotency reviewer claimed `_write_file` destroys user content after the managed marker. Refuted: the marker is an end-of-file freshness sentinel (v0.3.0 design), not a boundary; AGENTS.md is fully managed by intent. Pre-existing v0.3.6 behavior, not a v0.3.7 regression.
- Cross-agent reviewer escalated to BLOCK on default-on + listing MCP tool names by name. Refuted: owner explicitly chose default-on (per RFC Plan A); fallback text + `enabled=false` opt-out + `projects=[]` OSS safety net all in place. Concerns are legitimate UX polish (captured as v0.3.8 backlog), not BLOCK material.

**P1 items folded into the v0.3.7 commit**:
- Fallback prose: name baseline, prohibit retry/block, warn cold-start
- Asymmetry documentation: CHANGELOG explicitly explains why `[memory]` defaults on while `[promotion]` defaults off

**P2 items deferred to v0.3.8 backlog**:
- "MCP-accessible from any agent" template phrase is over-broad; should read "MCP-accessible from agents with `basic-memory` MCP registered"
- Silent exception swallow in `_build_memory_section`'s config-load path is graceful but hides BM-misconfig vs BM-disabled distinction; debug-level log would help operator diagnosis

## Migration for existing 12+ Theone workspaces

Each workspace's AGENTS.md needs to regenerate to pick up the new section. Two paths:

1. **Natural**: next time you run `vibe sync` (or `vibe start`) in each workspace, `refresh_adapters()` regenerates AGENTS.md with the new section automatically. No special command.
2. **Sweep** (optional): run a one-time loop across all workspaces to force regeneration without working in each:

```powershell
foreach ($p in Get-ChildItem 'D:\00Vibe_Coding' -Directory) {
    $vibe = Join-Path $p.FullName '.vibe'
    if (Test-Path $vibe) {
        Push-Location $p.FullName
        try { vibe sync 2>&1 } catch {}
        Pop-Location
    }
}
```

(Sync was preferred over start in the v0.3.6 sweep because it works in ACTIVE lifecycle and doesn't require state transition.)

## What this means for next-RFC-cycle

The `[[cross-team-dispatch-via-rfc]]` pattern is now confirmed as a portable methodology with two mandatory parts:

1. **Dispatch**: write the full RFC into target repo's `docs/`, do not commit code, do not modify existing files outside the new doc.
2. **Response**: when the target repo's owner adopts/adjusts/rejects, write a corresponding `*-RESPONSE.md` document next to the original RFC. **This is not optional** — without it, the dispatching session has no closure signal.

Both files together form the audit trail. Future contributors (human or AI) reading the repo can reconstruct the decision chain without needing access to either dispatch or response session's conversation history.

## What this release does NOT do

Documented for the next AI session that picks up `vibe-state-cli`:

- Does NOT change `[promotion]` defaults (still `enabled = false`).
- Does NOT add a 10th adapter (still 9, per v0.3.6's anti-pattern rejection).
- Does NOT modify the post-commit hook (v0.3.6's two-file-loop fix stays; the activity-log-skip backlog item carries forward).
- Does NOT auto-detect Basic Memory presence at adapter run time (Plan C in the RFC was rejected as added complexity for no real benefit given Theone's setup is consistent).
- Does NOT parameterize Basic Memory CLI path or MCP server name (everything stays implicit via the `mcp__basic-memory__*` convention).
- Does NOT update the existing 12 workspaces automatically on `pipx upgrade` — user must run `vibe sync` in each (this is by design; vibe never reaches into user repos without an explicit command).
