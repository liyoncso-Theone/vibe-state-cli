# RFC: AGENTS.md adapter — BM-aware Session Start Protocol

**Status**: Open for review
**Filed**: 2026-06-10 by laptop-side Claude session (for Theone)
**Target**: vibe-state-cli v0.3.6 (or later)
**Related**: [[v0.3.6-roadmap-RFC]] (the previous, broader v0.3.6 RFC)

---

## TL;DR

Add a `## Persistent Knowledge — QUERY BEFORE RECALL` section to the `agents_md` adapter's AGENTS.md template. Tells every agent (Claude, Codex, future) to query Basic Memory before answering recall questions. **Higher leverage than Claude-only `basicMemory` block in `.claude/settings.json`** — works across all agents that read AGENTS.md, requires no hook mechanism, can't time out.

---

## Context

On 2026-06-10, spent ~8 workflow iterations debugging why BM `SessionStart` hook brief wasn't reaching Claude Code in VS Code on Windows. Root cause: BM CLI cold-start 12s × 4 concurrent calls ≈ 31s, exceeds Claude Code's 30s default hook timeout → CC issues `hook_cancelled` → brief never reaches model. A 3-layer local workaround restored the brief (see BM `personal/bm-sessionstart-hook-windows-fix-2026-06-10`).

**During the post-mortem**, a bigger architectural insight surfaced:

1. Theone collaborates with **multiple agents** long-term (Claude + Codex + future tools). A Claude-only fix violates the multi-agent reality.
2. vibe-state-cli's `agents_md` adapter **already** produces an AGENTS.md template with a `## Session Start — READ THESE FILES` section instructing agents to read `.vibe/state/{current,tasks,standards}.md` on session open.
3. This is effectively the **cross-agent equivalent of a SessionStart hook** — pure markdown instructions, every agent that reads AGENTS.md follows, no hooks/timeouts/IDE-specific machinery required.
4. But the current template **does not mention Basic Memory at all**. So even though the BM data layer is universal (MCP-accessible from any client), agents reading the AGENTS.md don't know it's there.

This RFC closes that gap.

## Proposed Solutions

### Plan A — Default-on, with naming abstraction (recommended)

Add this section to the AGENTS.md template, **after** the existing `## Session Start — READ THESE FILES` and **before** `## Workflow`:

```markdown
## Persistent Knowledge — QUERY BEFORE RECALL

This project's persistent cross-machine knowledge is in **Basic Memory** (markdown-on-disk knowledge graph, accessible via MCP).

Before answering recall questions ("what did we decide", "where did we leave off", "what's our principle on X"), query the knowledge layer first:

- `mcp__basic-memory__search_notes(query="…", project="personal")` — fast text + semantic search
- `mcp__basic-memory__search_notes(metadata_filters={"type": "decision"}, project="methodology")` — structured query by type
- `mcp__basic-memory__build_context(url="memory://<topic>", depth=2)` — graph traversal from a seed note

Projects to read:
- `personal` — identity, machine-specific facts, preferences
- `methodology` — decision-style principles, feedback, coordination patterns

Capture material decisions as notes with `type: decision` to keep the layer fresh.
```

**Why default-on**: BM is the assumed knowledge layer for Theone's setup. New users without BM will either (a) ignore the section, or (b) be prompted to set up BM, which is the right outcome.

**Why "Basic Memory" not "second brain"**: concrete tool name is unambiguous and aligns with current MCP tool naming (`mcp__basic-memory__*`). If BM is later replaced by another backend, only this template changes — every existing AGENTS.md regenerates on `vibe sync`.

### Plan B — Opt-in flag

Make it triggered via `vibe adapt --add agents-md --with-basic-memory` (or similar). Useful if you want explicit opt-in semantics consistent with current adapter activation pattern.

**Trade-off**: extra cognitive overhead per workspace. Most Theone-owned workspaces would want this anyway.

### Plan C — Auto-detect

Detect BM presence at adapter run time (check for `~/.basic-memory` dir or `basic-memory` CLI on PATH); inject BM section only if detected.

**Trade-off**: adds adapter-time branching. Probably not worth it given Theone's setup is consistent across machines.

---

## Recommended: Plan A

Default-on, no flag, name = "Basic Memory" (concrete). Future flexibility comes from template editing, not config branching.

## Breaking Changes

- **None to existing functionality**.
- Existing AGENTS.md files don't auto-update — user must run `vibe sync` (or `vibe adapt`?) to regenerate. Need to confirm which command refreshes AGENTS.md.
- **Detect freshness**: per `CHANGELOG.md` line 100, the freshness detector examines all managed files for the current summary marker; adding a new section means the detector should still pass (marker is preserved). Verify in test plan.

---

## Migration Path

**New workspaces** (after this RFC lands): `vibe init` → AGENTS.md already includes BM section. Zero user action required.

**Existing workspaces** (Theone's 12 vibe-managed projects): User runs `vibe sync` (or `vibe adapt --refresh agents-md` if that command exists) → AGENTS.md regenerates with new section. Existing user content (non-managed sections) preserved.

**Workspaces without BM installed**: Section still injected, but it points to an MCP server that doesn't exist. Agent will fail the query and likely warn user — surfaces the missing dependency rather than silently degrading.

## Open Questions

1. **Naming**: "Basic Memory" (concrete) or abstract "second brain" / "knowledge layer"? Recommendation: keep "Basic Memory" for now, abstract later if BM is replaced.
2. **Which command regenerates AGENTS.md?** `vibe sync`? `vibe adapt`? Worth documenting clearly in CHANGELOG.
3. **Project names hardcoded?** Currently the proposal mentions `personal` and `methodology` projects by name. These are Theone-specific. Should the template parameterize them via vibe config? E.g. `vibe.toml`: `bm.primary_project = "personal"`, `bm.secondary_projects = ["methodology"]`. Then template substitutes at render time.
4. **Should the section link to specific BM notes?** E.g., `[[methodology/feedback-vscode-is-the-baseline]]`. Probably not — keeps AGENTS.md tool-agnostic and avoids stale links.
5. **What if BM is offline / MCP not registered?** Agent fails the query and falls back to "no memory available". Should template explicitly say "if MCP basic-memory is unavailable, proceed without it"? Or trust agent's own error handling?
6. **Should `claude` adapter's existing rules/standards have a corresponding update?** Probably yes — but lower priority, since AGENTS.md is already loaded by claude.
