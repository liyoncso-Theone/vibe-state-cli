## 2. CRITICAL BUG: Two tracked state files mutate on every commit → infinite loop

### Reproduction (deterministic)

```bash
cd <any-vibe-initialized-repo>
git commit --allow-empty -m "test"
git status
# Expected output (verified in vibe-state-cli's own repo at v0.3.5):
#   modified: .vibe/state/.sync-cursor
#   modified: .vibe/state/current.md
git commit -am "stash state changes"
git status
# Again shows the same two files modified.
# Loop continues forever.
```

### Root cause

Both files are **`git ls-files`-tracked** AND **rewritten by the post-commit hook** every time. The hook fires `vibe sync --no-refresh`, which:
1. Updates `.vibe/state/.sync-cursor` with the current `HEAD` SHA
2. Appends a `## Sync <timestamp>` block to `.vibe/state/current.md`

Both mutations land in the working tree as "modified, not staged." Committing them re-fires the post-commit hook, re-mutating the same two files — infinite loop with O(1) growth per cycle.

### Design intent vs. design oversight

The intent — preserving "where we last synced" and "recent activity summary" across `git clone` — is correct. The oversight is putting two attributes on the same file:
- **mutates on every commit** (machine-driven)
- **git-tracked** (user-content semantics)

These two attributes are incompatible by `git`'s definition.

The Unix convention (also Anthropic's Claude Code session model with `~/.claude/projects/<hash>/*.jsonl`): **runtime state is untracked; user-curated content is tracked. They live in separate files.**

### Recommended fix (Option A — minimum code, cleanest semantics)

Two file moves, one hook behavior change, plus migration on `vibe init --force`:

| File | Current | After |
|---|---|---|
| `.vibe/state/.sync-cursor` | tracked | **untracked** (add to `.vibe/.gitignore`, `git rm --cached`) |
| `.vibe/state/.lifecycle` | tracked | **untracked** (same — it's FSM state, not user content) |
| `.vibe/state/.hook.log` | already untracked | unchanged |
| `.vibe/state/current.md` | tracked, auto-appended by hook | **tracked**, but **only mutated by explicit `vibe sync` / `vibe start`** — post-commit hook stops appending |
| `.vibe/state/tasks.md`, `standards.md`, etc. | tracked | unchanged |

#### Hook behavior change

Current: post-commit hook calls `vibe sync --no-refresh`, which both updates `.sync-cursor` and appends to `current.md`.

Proposed: split into two paths.
- `vibe sync --no-refresh` (called from post-commit hook): updates **only** `.sync-cursor` and `.hook.log` — never touches `current.md`.
- `vibe sync` (no flag, user-invoked) and `vibe start` (already user-invoked): updates `.sync-cursor` **and** appends to `current.md`.

This preserves the "AI reads `current.md` for recent activity" feature when the user **deliberately** asks for a sync, while eliminating the commit-loop entirely.

#### Migration on `vibe init --force` and `vibe start`

Idempotent helper (mirror of `ensure_internal_gitignore()` pattern shipped in v0.3.5):

```python
# pseudocode
def ensure_state_files_untracked(repo_root: Path) -> None:
    """v0.3.6 migration: move .sync-cursor and .lifecycle out of git tracking."""
    needs_ignore = ["state/.sync-cursor", "state/.lifecycle"]
    ensure_internal_gitignore_appends(needs_ignore)
    for relpath in needs_ignore:
        if path_is_tracked(relpath):
            run("git", "rm", "--cached", relpath, "--quiet")
    # File contents stay on disk — only the index entry is removed.
```

Run this from both `vibe init --force` and `vibe start` so existing projects upgrade silently.

### Rejected alternatives (with reasoning, for completeness)

| Option | Reason rejected |
|---|---|
| **B**: Hook auto-amends commit (`git commit --amend --no-verify`) to absorb state changes | Rewrites recent commits; breaks after `git push`; surprises CI and `git pull --rebase` users. Strongly anti-pattern. |
| **C**: Move from `post-commit` hook to `pre-push` hook (fewer triggers) | Loop becomes rarer but doesn't disappear. Semantic mismatch — sync should follow commit, not push. |

### Test plan

```python
def test_post_commit_hook_does_not_modify_current_md(tmp_git_repo):
    init_vibe(tmp_git_repo)
    initial_current_md_hash = file_hash(tmp_git_repo / ".vibe/state/current.md")

    assert file_hash(tmp_git_repo / ".vibe/state/current.md") == initial_current_md_hash
    assert "modified" not in run("git", "status", "--porcelain", cwd=tmp_git_repo).output

def test_explicit_vibe_sync_does_modify_current_md(tmp_git_repo):
    # Existing behavior preserved
    ...

def test_migration_untracks_sync_cursor(tmp_git_repo_with_old_layout):
    run_vibe_init_force(tmp_git_repo_with_old_layout)

    assert (tmp_git_repo_with_old_layout / ".vibe/state/.sync-cursor").exists()
```

## 3. v0.3.6 Scope (Recommended — in priority order)

### 3.1. Fix the two-file infinite loop (Section 2)
See above. Highest priority. **Driver: dogfood bug, completely independent of any external memory tool.**

### 3.2. Remove the unused MCP optional dependency

`pyproject.toml` line ~47:
```toml
mcp = [
    "mcp>=1.27.0",
]
```

`grep -rn "mcp\|fastmcp\|ModelContextProtocol" src/` returns zero hits. The dep advertises functionality that does not exist. **Remove the optional extra entirely.** Add one line to README clarifying "MCP integration is not currently shipped; planned but deferred. See issue #..." if any users have inquired (none filed as of 2026-06-07).

**Driver: dead-code cleanup, completely independent of any external memory tool.**

### 3.3. Pivot the canonical write target to `AGENTS.md`

**Context**: AGENTS.md is now a Linux Foundation Agentic AI Foundation anchor project (announced 2025-12), with native readers in Codex CLI, GitHub Copilot, Cursor, Windsurf, Amp, Devin, Jules, VS Code, Gemini CLI. 60,000+ repos adopted. Claude Code still doesn't auto-load AGENTS.md but the official workaround is a one-line `CLAUDE.md` containing `@AGENTS.md`.

**Proposal**: Make `agents_md` adapter the **canonical** writer. Adapters for AGENTS.md-aware tools (`cursor`, `windsurf`, `copilot`, etc.) emit a **one-line shim** that pulls in AGENTS.md instead of duplicating the 5-line summary.

**Net effect**: Less code per adapter, less drift surface, follows the standard.

**Driver: industry standard alignment, completely independent of any external memory tool.**

References:
- [Linux Foundation announcement](https://www.linuxfoundation.org/press/linux-foundation-announces-the-formation-of-the-agentic-ai-foundation)
- [OpenAI Codex AGENTS.md docs](https://developers.openai.com/codex/guides/agents-md)
- [Claude Code issue #34235 — feature request for native AGENTS.md support](https://github.com/anthropics/claude-code/issues/34235)

### 3.4. Sunset the `antigravity` adapter's `GEMINI.md` output (deadline: 2026-06-18)

Google is shutting down Gemini CLI on 2026-06-18, replacing with Antigravity CLI (`agy`). The new tool uses `AGENTS.md` and `.agents/` rather than `GEMINI.md` and `.gemini/`.

**Proposal**:
- `antigravity` adapter stops writing `GEMINI.md` after 2026-06-18.
- It writes `.agents/skills/<vibe-skills>` instead (matches new CLI).
- For backward compatibility during transition (now → 6/18), keep writing both, with a deprecation note in `CHANGELOG`.

**Driver: third-party vendor deprecation deadline, completely independent of any external memory tool.**

Reference: [Google Developers Blog](https://developers.googleblog.com/an-important-update-transitioning-gemini-cli-to-antigravity-cli/)

### 3.5. Add `vibe promote` command (generic verb, default backend = `basic-memory` CLI)

> **⚠️ Driver clarification**: This is **the only v0.3.6 item motivated by the owner's specific tool stack** (Basic Memory). However, the command is **designed extensibly** — the default backend is `basic-memory`, but the architecture accommodates Obsidian CLI, Logseq, or any markdown-writing tool via config. The command itself is value-positive even for users who don't use Basic Memory; they'd just configure a different backend or leave it disabled.

**Why**: Some users (specifically including the owner) use a cross-project knowledge store (Basic Memory in owner's case) for "methodology" / cross-project patterns. Currently the workflow is fully manual: noticing a `vibe sync --note "..."` block is reusable across projects, then copy-pasting into the methodology store.

**Proposal**: Add an **explicit, opt-in, human-triggered** verb that does this promotion via subprocess call, with zero hard dependency on Basic Memory.

```
$ vibe promote "checkpoint-based git auto-commit pattern"
# Opens $EDITOR pre-filled with the most recent `## Sync` block from current.md.
# User edits down to essence + adds rationale.
# Saves → vibe runs:
#   basic-memory tool write-note \
#     --project methodology \
#     --folder vibe-promotions \
#     --title "checkpoint-based git auto-commit pattern" \
#     --content - <<< "$EDITED_CONTENT"
# Note frontmatter includes: source-project, source-commit, promoted-at.
```

**Config in `.vibe/config.toml`** (opt-in):
```toml
[promotion]
enabled = false
target = "basic-memory"        # extensible: future targets might include obsidian, raw-file, etc.
project = "methodology"
folder = "vibe-promotions"
```

**Failure mode**: If `basic-memory` CLI not on PATH, print a friendly error pointing to install docs. **Do not** make `basic-memory` a Python dep. The external binary is the contract.

**Why this is the *right* shape** (vs. an adapter or auto-promotion):
- Explicit verb, not a daemon → matches user's "no scheduled disruptions" preference
- One-way (vibe → external store only) → no bidirectional coupling
- Subprocess to external CLI → vibe stays single-purpose
- Opt-in config → users who don't use the feature see zero new complexity
- Extensible to other targets later (Obsidian CLI, Logseq, etc.) without refactor

**~80 LOC + ~5 tests, ~1 day of work.**

### 3.6. Bump status: `Development Status :: 3 - Alpha` → `4 - Beta`

`pyproject.toml` classifiers. Vibe has been in production across 11+ projects for 60+ days. The "Alpha" label in PyPI metadata is now false advertising. Bumping to Beta signals to potential users "this is real, but we're not promising semver stability yet."

Not "Production/Stable" (`5`) — that's the post-1.0 commitment we explicitly want to defer.

**Driver: truth-in-advertising, completely independent of any external memory tool.**

### 3.7. (Optional, low-priority) Document `.vibe/` cross-machine pattern in `docs/USER-GUIDE.md`

Add a one-page section: "Sharing `.vibe/` across machines."

Recommended primitives (in order):
1. **Git** (the default): `.vibe/` is already a tracked subdirectory. Just `git pull` / `git push` like everything else.
2. **Syncthing**: For users who want real-time sync of `.vibe/` (e.g. between desktop and laptop on same project). Document the `.stignore` patterns (exclude `state/.sync-cursor`, `state/.lifecycle`, `state/.hook.log`, `state/*.lock`).
3. **Don't do**: Cross-machine "project mirror" adapters writing to external memory stores. The user can Syncthing the folder directly; bolt-on adapters increase scope without value.

## 4. ANTI-PATTERNS — Do NOT do these in v0.3.6 or v0.4.x

Each item below was specifically considered, evaluated, and rejected by adversarial review. Documenting here to save future contributors the rediscovery cost.

### 4.1. Do NOT add a `basic_memory` adapter

**What it would do**: vibe writes the per-project state_summary into `~/BasicMemory/<bm-project>/projects/<project-name>.md` so that AI sessions across machines see project state via Basic Memory's knowledge graph.

**Why rejected**:
1. **Violates vibe's model-agnostic positioning**. Currently every adapter is symmetric — they're all "an AI tool that reads its own config file." Adding a Basic-Memory-specific adapter privileges one specific knowledge-store product. The next obvious question is "what about Obsidian / Logseq / Notion / Reflect / Mem?" — and there's no clean answer.
2. **Violates `AdapterBase`'s implicit contract** that `emit()` writes inside `project_root`. A BM adapter writes to `~/BasicMemory/...` — the first cross-boundary adapter. This is a structural change that opens future adapters to write anywhere, hurting predictability.
3. **The use case is better served by Syncthing-ing `.vibe/` directly** (see 3.7). Same problem, zero new code.
4. **For cross-project promotion** — which is the part that *does* belong in BM — use `vibe promote` (Section 3.5) as an explicit verb instead of an automatic adapter.

### 4.2. Do NOT auto-pull from Basic Memory in `vibe start`

**What it would do**: When the user runs `vibe start`, vibe queries Basic Memory for "what does this user prefer for this kind of project" and prints a briefing to terminal.

**Why rejected**:
1. **Wrong consumer**. The AI tool (Claude / Cursor / Codex) reads BM via MCP directly. Printing a static digest to terminal puts the data in front of the human, but Claude doesn't see terminal stdout unless the user manually pastes.
2. **Wrong direction**. Methodology should flow *in conversation* via MCP search, not *at session start* via static injection. Static injection becomes stale and noisy.
3. **Breaks `vibe start`'s reliability profile**. Today it's offline-pure and runs in <1s. Adding a network/IPC call introduces a failure mode that fires on every session.

### 4.3. Do NOT add a 10th adapter

**Context**: vibe ships 9 adapters: `agents_md`, `claude`, `cursor`, `windsurf`, `cline`, `copilot`, `roo`, `antigravity`. That's already at the high end for a solo-maintained OSS project.

**Why rejected**:
1. **Maintenance cliff**. Each new adapter = format-coupling to that vendor + tests + docs + handling vendor schema changes (which happen quarterly in this space). The realistic upper bound for a solo maintainer in a fast-moving ecosystem is ~5–8 adapters before maintenance lag visibly degrades quality. vibe is already at 9.
2. **The right move is fewer adapters, not more.** The Section 3.3 pivot (AGENTS.md-canonical with shims) reduces *effective* adapter complexity. Adding a 10th moves in the wrong direction.
3. **New tools should adopt AGENTS.md**. If a new tool (Cody, Continue, OpenCode, etc.) ships *without* AGENTS.md support, the right response is to file an upstream feature request, not to write a vibe adapter that papers over their non-compliance.

### 4.4. Do NOT chase the "Cursor rules → Cursor skills" pivot

Sourced from a single dev.to comment, not confirmed by Cursor's official docs as of 2026-06-07. Wait for Cursor's official announcement before touching the `cursor` adapter.

### 4.5. Do NOT write an MCP server (yet)

The market signal is mixed (multiple competing memory MCPs, no consolidation). The file-writing approach is vibe's moat. Revisit only if (a) AGENTS.md migration is complete and (b) a clear MCP pattern emerges (e.g., Anthropic blesses one).

## 5. After v0.3.6 — Freeze 90 days

Once v0.3.6 ships, **commit no source code for 90 days**. Pure observation period.

### Hard stop signals (any one = extend freeze indefinitely)

- Claude Code ships first-party cross-tool memory sync. vibe pivots from "the synchronization tool" to "the legacy adapter shim" — reduce scope, don't expand.
- Another 30 days passes with zero user-filed bugs. Demand-side signal of product-market fit at current scope. Adding features now would chase imaginary users.
- A contributor proposes a 10th adapter. The maintenance-cliff threshold has been hit. Ship 1.0 and freeze the adapter list.

### Soft stop signals (any two = extend freeze)

- PyPI downloads plateau below 250/month for 60 days.
- Maintainer has not personally `git commit`ed to vibe in 60 days *while still using it daily* — the tool has reached "done."
- Maintainer finds themselves reaching for Basic Memory / their personal knowledge store instead of `vibe sync --note` for project notes. The use case has migrated; let vibe stay narrow.
